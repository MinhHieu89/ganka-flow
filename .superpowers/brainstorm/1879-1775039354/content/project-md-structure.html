PROJECT.md Structure

Choose how to organize the living project tracker

A

Module-First Structure

Organized by system module. Each module section has its own scope, business rules, status, and decisions.

1. Project Overview (clinic, team, tech stack)
2. Phase Roadmap (what's in/out per phase)
3. Modules
   ├── Patient Intake
   ├── Pre-Exam (Screening)
   ├── EMR (Doctor Exam)
   ├── Dry Eye System
   ├── Reporting
   ├── Pharmacy (future)
   ├── Optical Center (future)
   └── Online Sales (deferred)
4. Cross-Cutting Concerns (roles, data rules, notifications)
5. Key Decisions Log
6. Current Status

Pros

  • Easy to find module-specific rules
  • Maps to how developers work on features

Cons

  • Cross-module workflows (e.g. payment flow) split across sections
B

Phase-First Structure

Organized by implementation phase. Each phase has its modules, scope, and acceptance criteria.

1. Project Overview (clinic, team, tech stack)
2. System Architecture (modules, data rules, roles)
3. Phase 1: Clinical Core
   ├── Scope & modules
   ├── Business rules
   └── Acceptance criteria
4. Phase 2: Operations
   ├── Pharmacy, Optical Center, Payments
   └── ...
5. Phase 3: Online & Integration
6. Key Decisions Log
7. Current Status

Pros

  • Clear delivery roadmap
  • Easy to track "what's next"

Cons

  • Module rules scattered across phases
  • Harder to reference a single module's full spec
C

Hybrid: Overview + Modules + Phases

Modules define WHAT (business rules & scope). Phases define WHEN (delivery order). Best of both.

1. Project Overview
2. System Modules (full business rules per module)
3. Data & Architecture Rules
4. Roles & Permissions
5. Phase Roadmap (which modules/features per phase)
   ├── Phase 1: Clinical Core (current)
   ├── Phase 2: Operations
   └── Phase 3: Online & Integration
6. Key Decisions Log
7. Current Status & Next Steps

Pros

  • Module rules in one place
  • Delivery plan clearly separated
  • Recommended for living doc

Cons

  • Slightly longer document
